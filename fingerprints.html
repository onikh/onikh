FINGERPRINTS: A BRIEF EXPLANATION


For years, the internet was based on the insecure HTTP (Hypertext Transfer Protocol). However, as time has passed, the more secure HTTPS (Hypertext Transfer Protocol Secure) has become more prevalent. Also called SSL (Secure Sockets Layer) and TLS (Transport Layer Security), HTTPS was created for two primary reasons: privacy and security (Gibson). HTTPS ensures our privacy by encrypting data sent back and forth, improving our privacy and creating systems of authentication that ensure that the website you are visiting has a legitimate certificate proving its identity. A browser has a list of trusted CAs (Certificate Authorities), and if a website presents a certificate digitally signed by one of these authorities, the website is deemed secure and safe by the HTTPS protocol.

However, the advantages that come along with HTTPS can be negated by HTTPS proxy appliances, commonly used by schools, businesses, and other institutions. By adding a Pseudo Certificate authority to browsers or computers within their network, they can then bypass certification by appending another certificate authority (their own) to the list of trusted certificate authorities, allowing them to manipulate that certificate authority (Gibson). They then can create spoofed certificates to impersonate websites, and all input made by the user is not encrypted but goes through the proxy server. The data can then go through to the intended remote website as if there was a direct link to the remote website without an HTML proxy appliance. The browser and user both have no way of knowing, as everything is kept identical. These are a type of MITM (man-in-the-middle) exploit. By intercepting data and sending it to the intended source, one can essentially view all the data from the interaction, and the two parties exchanging data, they would have no way of knowing that their data is being intercepted (Fisher). Risks can include having data stolen, having data sent to remote websites be tampered with, and websites being modified (for example, running ads not placed by the website's owner).

Although HTTPS/SSL decryption cannot be prevented, the fact that certificates have unique private and public keys means that we can tell when one's data is being run through an SSL/HTML Proxy Appliance. Public keys, which everyone has access to, cannot be used to derive a private key, and what'the public keys encrypt only those with access to the private key can decrypt. An SSL/HTML proxy appliance does not have access to the private key, meaning that it must use a different public key for which it also has the private key, and from there communicate with the remote website (Gibson). This means that the certificate will have to have a different public key, allowing you to determine whether your traffic is being funneled through an SSL/HTML proxy appliance.

So how can we easily detect changes in the certificate of a website? Create a hash. Hashes are algorithms that take the data from the certificate as input and output a string of letters and numbers, which we can use as a fingerprint. What makes hashes such a useful tool is that if even one bit of information hashed is different from otherwise identical data input, the output hash will be completely different (on average half the characters change for a one-bit difference) (Gibson). Thus, no matter how similar the certificate is if it isnâ€™t the exact certificate of the remote website the fingerprint/hash will be drastically different. Different hash algorithms bring varied results. Some are better than others. For example, certificate fingerprints were created using the MD5 hashing algorithm in the past, but after the discovery of weaknesses in the algorithm, SHA1 has begun being used due to its strength and security. Ideally, a hash function doesn't give similar values for inputs (termed collisions), has low similarity between hash outputs that come from similar inputs, and has no patterns that can be discerned from their inputs (MD5).

However, these tests can not always be 100% accurate. For example, Google is a large conglomerate company employing thousands of websites and servers, many of which have different certificates. Some users would obtain different certificates, and those wishing to test the fingerprints against each other would be falsely led to believe that an HTML proxy appliance was in play, when in reality none is involved and the remote website simply gave them different certificates than the one they chose to compare it to (Gibson). For example, www.wordpress.com and wordpress.com have different certificates, despite being the same site.
